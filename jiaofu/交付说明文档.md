# 《我和AI一起学射箭》重构与交付说明

## 1. 我如何分析原项目

由于仅提供可执行文件、未提供源代码，采用了“黑盒逆向功能分析”方法，而不是反编译源码。

### 1.1 分析方法

1. **界面与流程观察**
- 启动原程序后，逐页记录功能模块、按钮行为、状态反馈、数据流转路径。
- 重点观察：姿态标注、模型训练、实时评分、数据采集四个核心页签。

2. **输出物与数据侧推断**
- 通过导入导出文件、打分结果、模型文件命名方式（`.pkl`）反推内部需要的输入输出协议。
- 对模型导入失败/成功样本做差异分析，确定兼容策略。

3. **对照截图与回归校验**
- 以原程序页面形态和用户反馈截图作为对照基线。
- 每轮改动后进行“视觉一致性 + 行为一致性”回归。

## 2. 我优先完成了哪些部分，为什么

### 2.1 第一优先级：功能闭环可用

先完成四大模块可跑通：
1. 姿态标注
2. 模型训练
3. 实时评分
4. 数据集采集

### 2.2 第二优先级：高频阻塞问题

优先修复影响用户持续使用的问题：
1. 模型导入不可用/按钮无响应
2. `.pkl` 导入失败
3. 导出成功但不下载
4. 实时评分延迟与无分数
5. 多人场景与上半身漏检导致标注不准

**原因：** 这些是用户在真实流程中第一时间遇到的“卡点”，不先解决会导致系统不可用。

### 2.3 第三优先级：交付与分发能力

1. 持续生成版本化 exe（v2...v23）
2. 可将代码+exe拷贝到交付目录
3. 支持双击 exe 自动打开浏览器

**原因：** 项目最终要“交付可运行版本”，而不仅是本地开发可运行。

## 3. 我放弃或延后的部分，以及原因

### 3.1 暂未做多人手动目标切换与关键点拖拽微调

当前多人图采用自动选主目标策略，未提供“手动切换目标人/手动微调点位”交互。

**原因：**
1. 需要新增较大交互面板与状态管理。
2. 优先级低于“稳定识别 + 可导出可导入 + exe分发”。

### 3.2 暂未做自动化测试体系（单测 + E2E）

目前主要依赖功能回归与人工验收，尚未形成系统化自动测试流水线。

**原因：**
1. 需求迭代频繁，先保证功能快速收敛。
2. 自动化测试建设成本高。

### 3.3 暂未做“完全无 Python 依赖”的 `.pkl` 兼容方案

当前外部 sklearn `.pkl` 转换路径仍可能依赖目标机 Python 环境。

**原因：**
1. 纯 Node 侧完整兼容 sklearn pickle 成本较高。
2. 交付时优先满足主流程可用，跨生态模型兼容作为下一阶段优化。

## 4. 如果再给半天到一天，我会继续做什么

### 4.1 半天到一天内

1. **标注页“目标人切换”**
- 在多人场景中提供“上一人/下一人”按钮，用户手动锁定目标。

2. **规则评分稳定性增强**
- 增加评分平滑、关键点缺失补偿、姿态覆盖率提示。

3. **导入导出诊断提示**
- 在界面上明确提示：模型来源、解析方式、失败原因建议。

4. **动作阶段评分（准备-拉弦-瞄准-放箭）**
- 总分之外增加分阶段评分，定位问题更精准。
